# 第四章：并发面试八股

> GCD / NSOperation 基础见 [OC 第四章](../OC/04-多线程面试八股.md)。本章侧重 **Swift Concurrency**（async/await、Actor、Sendable）。

## 4.1 三代并发方案

| 代际 | API | 特点 |
|------|-----|------|
| ① 线程 | `Thread`、`pthread` | 手动管理，易出错 |
| ② GCD / Operation | `DispatchQueue`、`OperationQueue` | 系统主流，C/OC 友好 |
| ③ Structured Concurrency | `async/await`、`Task`、`Actor` | 编译器检查、结构化取消 |

**面试结论**：新项目 Swift 代码优先 **Swift Concurrency**；与 OC/GCD 混用时用 `@MainActor` + `DispatchQueue.main` 桥接。

## 4.2 async / await 核心

```swift
func fetchUser() async throws -> User {
    let (data, _) = try await URLSession.shared.data(from: url)
    return try JSONDecoder().decode(User.self, from: data)
}

// 调用方
Task {
    do {
        let user = try await fetchUser()
        await MainActor.run { updateUI(user) }
    } catch {
        await MainActor.run { showError(error) }
    }
}
```

| 概念 | 说明 |
|------|------|
| `async` | 函数可挂起，不阻塞线程 |
| `await` | 挂起点，等待异步结果 |
| `Task` | 创建异步任务的工作单元 |
| 结构化 | Task 有父子关系，取消可传播 |

## 4.3 Actor（必背）

Actor 是 **引用类型**，保证内部状态 **串行访问**，解决数据竞争。

```swift
actor Counter {
    private var value = 0
    func increment() { value += 1 }
    func current() -> Int { value }
}

let counter = Counter()
Task {
    await counter.increment()  // 必须 await，跨 actor 边界
}
```

**与 lock 对比**：

| | Actor | NSLock / os_unfair_lock |
|--|-------|-------------------------|
| 安全 | 编译期 + 运行时 | 全靠程序员 |
| 死锁 | 较少（串行 executor） | 容易 |
| 性能 | 协调度，不阻塞线程 | 可能阻塞 |

**@MainActor**：标记 UI 相关类型/方法必须在主线程执行，替代「GCD 回主线程」。

```swift
@MainActor
class HomeViewModel: ObservableObject {
    @Published var items: [Item] = []

    func load() async {
        items = await api.fetchItems()  // 已在 MainActor，可直接改 UI 状态
    }
}
```

## 4.4 Sendable 协议

标记类型 **可安全跨并发域传递**。

| 自动 Sendable | 非 Sendable |
|---------------|-------------|
| 值类型（成员也 Sendable） | 普通 class（可变状态） |
| `actor` | 未保护的 global var |
| `@Sendable` 闭包 | 部分 `@unchecked Sendable` 需人工保证 |

```swift
// 编译器可能报错：class 非 Sendable
class Cache { var dict = [:] }

// 解法：actor 包装 / 改成 struct / @unchecked Sendable（谨慎）
```

**面试点**：Swift 6 语言模式下 Sendable 检查更严格。

## 4.5 Task 与取消

```swift
let task = Task {
    try Task.checkCancellation()
    return try await longRunning()
}

task.cancel()  // 协作式取消，需代码配合 checkCancellation
```

| API | 作用 |
|-----|------|
| `Task.detached` | 无父 Task，独立生命周期 |
| `TaskGroup` | 并行子任务，结构化 |
| `withTaskGroup` | 收集多个 async 结果 |
| `async let` | 并行绑定多个 await |

```swift
async let a = fetchA()
async let b = fetchB()
let (ra, rb) = try await (a, b)
```

## 4.7 与 GCD 对照

| GCD | Swift Concurrency |
|-----|-------------------|
| `dispatch_async(main)` | `await MainActor.run { }` / `@MainActor` |
| `DispatchGroup` | `withTaskGroup` / `TaskGroup` |
| `dispatch_once` | `static let`（懒加载线程安全） |
| 信号量限流 | `AsyncSemaphore`（第三方）或 actor 计数 |
| 回调 hell | `async/await` 线性代码 |

**混用注意**：

```swift
// GCD → async
await withCheckedContinuation { continuation in
    fetchLegacy { result in
        continuation.resume(returning: result)
    }
}

// async → GCD
DispatchQueue.global().async {
    Task { await doAsyncWork() }
}
```

## 4.8 死锁与线程安全（继承 OC 考点）

| 经典死锁 | 原因 |
|----------|------|
| 主队列 `sync` 自己 | 等自己执行完 |
| 串行队列 A sync B，B sync A | 互相等待 |

Swift Concurrency 下 **避免在主 actor 上 sync 等待** 仍是原则。详见 [OC 第四章 §4.4](../OC/04-多线程面试八股.md)。

## 4.9 高频面试 Q&A

### Q1：async/await 和 GCD 本质区别？

- GCD：基于 block 的线程/队列调度，无编译器并发安全检查
- async/await：基于协程，挂起不占线程，结构化生命周期 + Sendable 检查

### Q2：Actor 和 class + lock 哪个好？

Actor 语义更清晰、不易漏锁；高频热点锁可能仍用 `os_unfair_lock`。面试答：**业务状态用 Actor，极致性能用 lock**。

### Q3：@MainActor 和 DispatchQueue.main.async 区别？

@MainActor 是 **编译期标注**，调用跨边界需 `await`，类型系统保证 UI 隔离；GCD 纯运行时，无编译检查。

### Q4：Task 和 Thread 的区别？

Task 是 **协作式** 调度单元，挂起时不占 OS 线程；Thread 是 OS 级，1:1 或 N:M 映射，更重。
