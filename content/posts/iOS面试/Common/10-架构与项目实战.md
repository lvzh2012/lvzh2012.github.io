## 10.1 MVC vs MVVM：职责边界

### MVC 在 iOS 的问题

Apple 的 MVC 里 **Controller 往往过重**：
- 网络请求、解析、UI 更新、导航全在 VC
- VC 代码几千行 → 难测、难复用

### MVVM 核心思想

```
View（VC + View）  只负责展示和用户操作
       ↕ 绑定（单向或双向）
ViewModel          状态 + 业务逻辑 + 格式转换（不放 UIKit）
       ↕
Model              纯数据结构
       ↕
Service            网络/数据库
```

**ViewModel 不知道 View 的存在**——可单元测试（Mock ViewModel 测逻辑）。

### 绑定原理（几种实现）

| 方式 | 原理 |
|------|------|
| **KVO** | 观察 VM 属性 `@{@"title": ...}`，变化时 VC 更新 label |
| **Block/Closure** | VM 状态变 → 调 `onUpdate` 闭包 |
| **RAC/Combine** | 属性变 → Signal/Publisher → VC subscribe |
| **@Published** | SwiftUI/ObservableObject 自动刷新 View |

**数据流**：
```
用户点击 → VC 调 VM.load()
    → VM 调 Service 请求
    → 解析 Model
    → VM 更新 @Published items
    → View 自动/手动刷新
```

### 美团二面：省市区三级设计

```
ProvinceListVM
  ├── provinces: [Province]
  ├── selected: Province?
  └── fetchProvinces() → Service

CityListVM(provinceId)
  ├── cities: [City]
  └── ...

DistrictListVM(cityId)
  └── ...

Coordinator / Router 管理 VC1→VC2→VC3 跳转与参数
```

**返回上级**：Navigation pop；VM 不持有 VC，避免循环引用。

## 10.2 组件化 / 路由原理

### 为什么组件化？

多业务线（首页/直播/电商）同仓 → 编译慢、耦合高。拆成 **独立 Pod/Module**，只暴露 **协议接口**。

### CTMediator 原理（Target-Action）

```
Module A 想打开 Module B 的页面，但不 import B

CTMediator:
  performTarget:@"BModuleTarget"
         action:@"gotoDetail:"
         params:@{@"id": @123}

Runtime:
  NSClassFromString(@"BModuleTarget") → 实例化
  performSelector 调 gotoDetail:
  BModuleTarget 内部 push B 的 DetailVC
```

**本质**：Runtime **动态类名 + 方法名** 解耦，编译期 A 不依赖 B。

### URL Router 原理

```
myapp://live/room?id=123
    → AppDelegate openURL
    → Router 解析 path + query
    → 查路由表 { "live/room": LiveRoomHandler }
    → Handler 创建 VC 并 push
```

## 10.3 字典转 Model 原理

### KVC 方式

```objc
Person *p = [[Person alloc] init];
[p setValue:@"Tom" forKey:@"name"];  // 找 setName: 或直接设 ivar
```

**风险**：key 拼错运行时才发现；不存在的 key 可能 KVC 崩溃。

### Runtime 枚举属性（YYModel/MJExtension 思路）

```
1. class_copyPropertyList 拿到所有属性名、类型
2. 遍历 JSON key
3. 匹配属性名（可配置下划线转驼峰）
4. 按类型转换：
   - 基础类型 NSNumber → int/float
   - NSString → 直接赋
   - 嵌套对象 → 递归创建子 Model
   - 数组 → 泛型 class 映射 elementClass
5. KVC 或 objc_setAssociated 赋值
```

**Swift Codable 原理**：编译期生成 `init(from decoder:)`，**类型安全**，无 Runtime 开销——字节三面「为何不用 YYModel」的 Swift 答法。

## 10.4 夜间模式原理

```
ThemeManager.shared.current = .dark
    │
    ├─ 更新颜色 Token 表（primaryColor、backgroundColor…）
    ├─ post Notification / Combine send
    └─ 各 VC 监听 → 刷新 UI / 调 setNeedsDisplay

iOS 13+ 推荐：
  Asset Catalog 动态颜色 + traitCollectionDidChange
  系统自动跟随系统 Dark Mode
```

## 10.5 数据持久化选型原理

| 方案 | 底层 | 适用 |
|------|------|------|
| UserDefaults | plist 键值 | < 几百 KB 配置 |
| Keychain | 加密 SQLite | Token、密码 |
| 归档 | NSKeyedArchiver / Codable Data | 对象 blob |
| SQLite | B+ 树索引文件 | 结构化查询、大量数据 |
| Core Data | SQLite + 对象图 | 复杂关系、迁移 |

## 10.6 设计模式在 iOS 中的原理

| 模式 | 原理 | 典型 |
|------|------|------|
| **单例** | static 实例 + dispatch_once 保证唯一 | `shared` 网络层 |
| **观察者** | Subject 维护 Observer 列表，状态变 notify | KVO、Notification |
| **代理** | 反向调用，VC 不继承 TableView | weak delegate 防循环 |
| **策略** | 算法/行为封装成可替换对象 | 多种支付方式 |

**单例 vs 全局变量**：单例 **控制实例唯一** + 可懒加载 + 可 mock（测试替换成 fake）。
