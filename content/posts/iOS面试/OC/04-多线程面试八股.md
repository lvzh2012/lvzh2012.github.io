# 第四章：多线程面试八股

## 4.1 为什么需要多线程？

| 目的 | 说明 |
|------|------|
| **提升体验** | 耗时操作放子线程，主线程保持 UI 流畅（60fps ≈ 16.7ms/帧） |
| **充分利用 CPU** | 多核并行处理 |
| **同时处理 I/O** | 网络、磁盘读写不阻塞 UI |

**铁律**：**UI 更新必须在主线程**。

---

## 4.2 iOS 多线程方案对比

| 方案 | 特点 | 使用场景 |
|------|------|----------|
| **NSThread** | 轻量、手动管理生命周期 | 几乎不用，了解即可 |
| **GCD** | C API、轻量、系统底层 | **最常用**，异步任务 |
| **NSOperation** | 面向对象、可取消、依赖、KVO | 复杂任务编排 |
| **pthread** | POSIX 标准、C 语言 | 底层库，App 层少见 |

**面试结论**：日常 **GCD 为主**，需要取消/依赖/进度时用 **NSOperation**。

---

## 4.3 GCD 核心概念

### 队列类型

| 队列 | 说明 |
|------|------|
| **主队列** `dispatch_get_main_queue()` | 串行，主线程，UI 专用 |
| **全局并发队列** `dispatch_get_global_queue()` | 系统提供，并发 |
| **自定义串行队列** | 自己创建，任务顺序执行 |
| **自定义并发队列** | 自己创建，多任务并行 |

### 任务提交方式

| API | 行为 |
|-----|------|
| `dispatch_async` | **异步**，不阻塞当前线程 |
| `dispatch_sync` | **同步**，等任务执行完才返回 |

```objc
// 子线程做耗时，主线程刷 UI
dispatch_async(dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0), ^{
    NSData *data = [self loadData];
    dispatch_async(dispatch_get_main_queue(), ^{
        self.imageView.image = [UIImage imageWithData:data];
    });
});
```

### QoS 优先级（从高到低）

`userInteractive` → `userInitiated` → `default` → `utility` → `background`

## 4.4 GCD 死锁（必考 ⚠️）

### 经典死锁：主队列 sync

```objc
// ❌ 必死锁
dispatch_sync(dispatch_get_main_queue(), ^{
    // 主队列等主队列，互相等待
});
```

### 串行队列 sync 自己

```objc
dispatch_queue_t queue = dispatch_queue_create("com.test", DISPATCH_QUEUE_SERIAL);
dispatch_async(queue, ^{
    dispatch_sync(queue, ^{  // ❌ 同一串行队列 sync 自己 → 死锁
    });
});
```

### 死锁条件总结

> **当前队列** 对 **同一串行队列** 调用 `dispatch_sync`，且该 sync 任务需要等当前任务结束 → 互相等待。

### 并发队列 sync 自己

```objc
// ✅ 不会死锁，但会开新线程执行（不推荐这样写）
dispatch_queue_t concurrentQ = dispatch_queue_create("c", DISPATCH_QUEUE_CONCURRENT);
dispatch_async(concurrentQ, ^{
    dispatch_sync(concurrentQ, ^{ /* OK */ });
});
```

## 4.5 GCD 常用 API

| API | 用途 |
|-----|------|
| `dispatch_group` | 等多任务全部完成 |
| `dispatch_barrier` | 并发队列中「写屏障」，读写锁语义 |
| `dispatch_semaphore` | 信号量，控制并发数 / 线程同步 |
| `dispatch_once` | **单例** 线程安全初始化 |
| `dispatch_after` | 延迟执行（不保证精确时间） |
| `dispatch_apply` | 并发遍历（注意控制并发度） |

### dispatch_group 示例

```objc
dispatch_group_t group = dispatch_group_create();
dispatch_group_enter(group);
dispatch_async(queue, ^{
    // task A
    dispatch_group_leave(group);
});
dispatch_group_enter(group);
dispatch_async(queue, ^{
    // task B
    dispatch_group_leave(group);
});
dispatch_group_notify(group, dispatch_get_main_queue(), ^{
    // A、B 都完成，回主线程
});
```

### dispatch_once 单例

```objc
+ (instancetype)shared {
    static id instance = nil;
    static dispatch_once_t onceToken;
    dispatch_once(&onceToken, ^{
        instance = [[self alloc] init];
    });
    return instance;
}
```

### dispatch_semaphore 限制并发数

```objc
dispatch_semaphore_t sem = dispatch_semaphore_create(3); // 最多 3 个并发
dispatch_semaphore_wait(sem, DISPATCH_TIME_FOREVER);
// 执行任务
dispatch_semaphore_signal(sem);
```

## 4.6 NSOperation / NSOperationQueue

| 特性 | GCD | NSOperation |
|------|-----|-------------|
| 取消任务 | 难 | ✅ `cancel` |
| 任务依赖 | 需 group 等 | ✅ `addDependency:` |
| KVO 进度 | 无 | ✅ `NSProgress` |
| 优先级 | QoS | `queuePriority` |
| 最大并发 | 队列控制 | `maxConcurrentOperationCount` |

```objc
NSBlockOperation *op1 = [NSBlockOperation blockOperationWithBlock:^{ /* A */ }];
NSBlockOperation *op2 = [NSBlockOperation blockOperationWithBlock:^{ /* B */ }];
[op2 addDependency:op1];  // B 等 A 完成

NSOperationQueue *queue = [[NSOperationQueue alloc] init];
[queue addOperations:@[op1, op2] waitUntilFinished:NO];
```

**底层**：NSOperationQueue 通常 **基于 GCD 实现**。

## 4.7 线程安全

### 线程安全问题类型

| 问题 | 说明 |
|------|------|
| **竞态条件** | 多线程同时读写，结果依赖执行顺序 |
| **死锁** | 互相等待锁 |
| **优先级反转** | 低优先级持锁，高优先级等待 |

### 常见锁

| 锁 | 特点 |
|----|------|
| `@synchronized(obj)` | 简单，性能一般 |
| **`NSLock` / `NSRecursiveLock`** | 递归锁可重入 |
| **`pthread_mutex`** | 底层互斥锁 |
| **`OSSpinLock`** | 自旋锁，**已废弃**（优先级反转） |
| **`os_unfair_lock`** | 替代 SpinLock，轻量 |
| **`dispatch_semaphore`** | 信号量 |
| **`dispatch_barrier`** | 多读单写 |
| **`@atomic`** | 只保证 setter/getter **原子**，不保证整体线程安全 |

### atomic vs nonatomic

```objc
@property (atomic, strong) NSMutableArray *list;
// atomic：getter/setter 单次读写原子
// 但 thread1 读 + thread2 写 + thread1 改 仍不安全！
// 线程安全需：串行队列 / 锁 / 不可变设计
```

**面试标准答法**：

> `atomic` 只保证 **属性赋值操作** 不会被线程打断，**不等于** 对象内部业务逻辑线程安全。

## 4.8 多线程 + RunLoop

> RunLoop 与触摸事件的关系见 [第五章 扩展八股](./05-扩展八股.md)、[Common 第六章 Hit-Testing](../Common/06-事件传递链HitTesting.md)。

| 线程 | RunLoop |
|------|---------|
| **主线程** | 默认开启，处理 UI、Timer、Source |
| **子线程** | **默认没有**，需要手动 `run`（如常驻线程） |

```objc
// 常驻线程典型写法
- (void)threadEntry {
    @autoreleasepool {
        NSRunLoop *runLoop = [NSRunLoop currentRunLoop];
        [runLoop addPort:[NSPort port] forMode:NSDefaultRunLoopMode];
        [runLoop run];  // 不会返回，保持线程存活
    }
}
```

**关联**：

- `performSelector:onThread:` 目标线程需 RunLoop
- `NSTimer` 依赖 RunLoop
- 网络回调线程取决于 API 设计（NSURLSession delegate 队列可指定）

## 4.9 常见坑

| 坑 | 说明 |
|----|------|
| 子线程更新 UI | 崩溃或未定义行为，必须 `dispatch_async(main)` |
| 主队列 sync | 死锁 |
| `dispatch_after` 当 Timer | 不精确，Block 捕获可能循环引用 |
| 过度创建线程 | GCD 有线程池，不必自己 NSThread 满天飞 |
| 锁粒度太大 | 性能差；太小可能死锁 |
| `@synchronized(self)` 锁 self | self 被多处锁可能死锁，建议专用 token 对象 |

## 4.10 面试 Q&A

### Q1：GCD 串行 vs 并发队列？

- **串行**：同一时刻只执行一个任务，顺序 FIFO
- **并发**：可多个任务同时执行（由 GCD 线程池调度）

### Q2：`dispatch_async(main)` 一定在主线程吗？

调用时 **已在主线程** → 可能同步执行 block（不新开线程）；  
在 **子线程** 调用 → 异步抛到主队列。

### Q3：如何实现线程安全的单例？

`dispatch_once` — 系统保证 block 只执行一次，且线程安全。

### Q4：读写锁怎么实现？

并发队列 + `dispatch_barrier_async`：

- 读：`dispatch_async` 并发读
- 写：`dispatch_barrier_async` 写时独占

### Q5：GCD 和 NSOperation 怎么选？

- 简单异步 → GCD
- 需取消、依赖、进度 → NSOperation
- AFNetworking 早期用 NSOperation，现多用 GCD + 封装

### Q6：以下会死锁吗？

```objc
dispatch_queue_t q = dispatch_queue_create("s", DISPATCH_QUEUE_SERIAL);
dispatch_async(q, ^{
    NSLog(@"1");
    dispatch_sync(q, ^{
        NSLog(@"2");  // ❌ 死锁
    });
});
```
